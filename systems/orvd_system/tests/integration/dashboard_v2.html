Центральный компонент. Авторизует взлёт, проверяет сертификаты, обрабатывает регистрацию. Защита от replay-атак.

Цели безопасности
Security Goals согласно ГОСТ Р 72118-2025 и методологии KasperskyOS
SG-01
Аутентификация компонентов
Только авторизованные компоненты могут взаимодействовать с системой. Каждый компонент имеет уникальный идентификатор, верифицируемый монитором.
✓ Реализовано
SG-02
Контроль периметра
Все внешние взаимодействия проходят только через Gateway. Прямой доступ к внутренним компонентам исключён на уровне сетевой изоляции Docker.
✓ Реализовано
SG-03
Авторизация полётов
Дрон не может взлететь без явной авторизации от ORVD Component. Проверяются запретные зоны, сертификат регулятора и состояние миссии.
✓ Реализовано
SG-04
Изоляция недоверенных доменов
Телеметрия и внешние данные физически изолированы от доверенного ядра. Контейнерная изоляция + политика брокера не допускают прямой связи.
✓ Реализовано
SG-05
Аудит и журналирование
Все значимые события записываются в ORVD Journal. Журнал изолирован — компоненты могут только добавлять записи, но не изменять существующие.
✓ Реализовано
SG-06
Мониторинг безопасности
Security Monitor перехватывает все сообщения и применяет политику allow-list. Сейчас логика встроена в компоненты — планируется вынести в отдельный контейнер.
⚠ В работе
Изоляция доменов
Как контролируется взаимодействие между доверенным и недоверенным доменами
Поток авторизации взлёта
1
Запрос от оператора
Оператор регистрирует дрон и миссию через Gateway. ORVD Component — единственный авторизующий компонент.
register_drone → register_mission → authorize_mission
2
Перехват монитором
Монитор перехватывает сообщение в брокере и проверяет политику allow-list.
Monitor: check_policy(action="register_drone") → ALLOW
3
Проверка запретных зон
ORVD Component проверяет маршрут миссии. Если пересекает зону — отклоняет.
register_mission: route intersects no_fly_zone? → rejected / mission_registered
4
Запрос взлёта от дрона
Дрон отправляет запрос с mission_id. ORVD Component проверяет авторизацию миссии.
request_takeoff(drone_id, mission_id) → takeoff_authorized / denied
5
Выдача авторизации
ORVD Component выдаёт авторизацию. Событие записывается в Journal. Телеметрия начинает мониторинг.
ordv/authorization: {"authorized": true} + journal.write(flight_auth_success)
Четыре уровня изоляции
1. Сетевая изоляция
Docker networks разделяют домены. Недоверенные контейнеры не имеют прямого доступа к доверенным.
2. Брокер как посредник
Все коммуникации через Kafka/MQTT. Прямые TCP-соединения между компонентами запрещены конфигурацией.
3. Политика allow-list
Монитор применяет явную политику: запрещено всё, что не разрешено явно. Каждое правило привязано к SG.
4. Аудит всех событий
ORVD Journal фиксирует все события. Попытки нарушения политики логируются и инициируют тревогу.
